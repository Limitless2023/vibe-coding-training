一、我与 Agent 的协作模式

在 5 天的开发过程中，我和 Agent 之间的协作远不只是"提需求 → 写代码"。以下是我总结出的 7 种协作模式，每一种都对应了实际工作中的一个真实场景。

模式 A

让 Agent 构思优化方向

"你构思一下还有这个 feature 还有哪些需要优化"

场景：一个功能的主体逻辑已经完成，但我知道还有优化空间，只是自己不确定具体从哪些维度去改进。

做法：把当前实现的截图或代码丢给 Agent，让它站在"开发者 + 用户"的双重视角去发散。

真实案例
  • 工具调用展示（P1 → P1.5 → P1.6 → P1.7）：P1 做完自然语言化后，我问 Agent "还有哪些可以打磨的"。它提出了字号层级统一（14/13/12px）、Badge 从实心改描边、左侧图标对齐、竖线厚度统一——这些我自己大概率想不到这么细。
  • 条款推荐卡片（P15.1）：基础功能做完后，Agent 提出了信息架构调整（Pills 分类标签）、配色分级（teal/blue 按复杂度）、内容预览截断、复制功能等优化方向，最终迭代了 10+ 轮。
  • 追问功能（P9）：引用追问做完后，Agent 建议了高亮持久化、5 状态状态机、浮动岛动画等增强。
价值：Agent 能从代码细节、用户体验、视觉一致性等多个维度系统性地提出优化建议，比我自己"想到哪改到哪"全面得多。
模式 B

让 Agent 批判性审查

"批判性地看一下目前的实现方式和效果，别搞成代码屎山"

场景：功能做了好几轮迭代后，担心代码质量下降——特别是 AI 生成的代码，容易出现重复逻辑、过度工程、命名混乱等问题。

做法：让 Agent 以"代码审查者"的视角重新审视当前实现，重点关注可维护性、一致性和是否有不必要的复杂度。

真实案例
  • 工具描述系统（P5）：Purpose 参数跨 3 仓库注入后，让 Agent 审查 tool-descriptions.ts 的模板 fallback 逻辑。Agent 发现有些模板和 purpose 的职责边界模糊，帮我理清了三级优先级：purpose 优先 → 工具模板 fallback → 默认描述。
  • Sub-Agent 事件流（P10~P17）：跨 3 仓库改完后，让 Agent 检查事件流的命名一致性。发现 meagent 层用 subagent_start、copilot 层转成 CUSTOM 事件类型、前端解析时又用了不同的 key。Agent 统一了命名规范。
  • Tooltip 样式：多轮 UI 迭代后，发现 tooltip 的实现方式零散——有的用原生 title，有的用组件库 Tooltip，有的用自定义 CSS。让 Agent 统一收敛。
价值：AI 生成的代码需要"被审查"，而 Agent 自己就能做这个审查——关键是你要主动触发这个模式，而不是默认它写出来的就是最优解。
模式 C

让 Agent 从用户角度思考

"从用户使用的角度，还可以做哪些 feature 或打磨"

场景：技术实现没问题了，但我想确保从"使用者"的视角来看体验也是好的。

做法：让 Agent 角色扮演一个普通用户，走一遍使用流程，找出体验上的问题或缺失。

真实案例
  • Composer 输入体验：功能都完成后，我让 Agent 从用户角度看还有什么不舒服的。Agent 提出了"发送后输入框应该自动聚焦"（省掉一次点击）和"拖放文件时应该有视觉反馈"。最终用纯 CSS 实现了拖放反馈，零 JS 状态管理。
  • 智能推荐问题（P10）：Agent 指出"推荐问题如果太长会溢出"——加了截断和 tooltip。还提出"推荐的问题应该和当前对话相关，不是通用问题"——调整了 LLM prompt。
  • Artifact 预览（P9）：Agent 建议增加"下载"和"新标签页打开"按钮——用户生成了一个漂亮的 HTML 报表，当然想保存下来。
价值：开发者（包括 AI）容易陷入"功能完成 = 体验完成"的思维，主动让 Agent 切换到用户视角能发现很多"最后一公里"的体验问题。
模式 D

让 Agent 用大白话解释

"没看懂，大变白话给我解释一下"

场景：Agent 给出的技术方案或代码解释太专业了，我需要它用更直白的方式讲清楚，以便我做出正确的判断。

做法：直接打断说"没看懂"，让它重新解释。不要假装理解——你的判断力建立在理解之上。

真实案例
  • AG-UI SSE 事件流：Agent 解释子 Agent 可视化方案时涉及 ContextVar、SSE 流、事件队列等概念。让它用大白话重说："相当于子 Agent 运行时往一个公共信箱里塞纸条，主进程不停地检查信箱把纸条转发给前端，前端收到纸条就更新界面。"
  • Pydantic create_model 的坑：大白话版："就像填表格时，以下划线开头的字段被认为是'内部备注'，不会出现在正式表单里。"
  • sandbox iframe 安全模型：大白话版："sandbox 像一个玻璃房间，allow-scripts 让它能动但不能出来，如果再加 allow-same-origin 就等于给了钥匙，它就能随便翻你的东西了。"
价值：你不需要懂代码，但需要懂"这件事的本质是什么"。让 Agent 用大白话解释，是 vibe coder 最重要的技能之一——理解了才能做对决策。
模式 E

先讨论，不行动

"我们先讨论，不行动"

场景：面对一个较大或不确定的改动，我想先和 Agent 对齐思路，确保方向正确后再动手。

做法：明确告诉 Agent "先不要改代码"，进入纯讨论模式。

工具层面的落地

我在 Claude Code 中会切到 Explore mode（只读模式，Agent 只能搜索和阅读代码，不能修改），讨论清楚后再切回执行。

强烈建议一定要开 Plan mode——不只是大功能，哪怕是中等规模的改动也建议开。Agent 出了 plan 之后，plan 一定要 review，确认每一步改什么、改哪些文件、改动范围是否合理，再放手让它执行。

不 review plan 就直接执行，跟不看需求文档就开始写代码没区别——方向跑偏了后面全是返工。

真实案例
  • 子 Agent 可视化方案选择：这是整个项目中最复杂的 feature。我先让 Agent 出 3 个方案（纯前端轮询 / 后端事件流 / WebSocket 双工），然后逐个讨论优劣。讨论了大概 20 分钟，确认方案 B 后才开始动手。
  • P2 渐变效果讨论：Composer 和 Welcome 组件的渐变色调了好几轮都不满意。最终决定"周一找擦神讨论"——知道什么时候该停也是协作的一部分。
  • 头像功能全栈方案：前端裁剪 + MinIO 存储 + Server DataItem，涉及 3 层。先和 Agent 讨论数据流转和接口设计，对齐后才开始编码。
价值：最贵的错误是方向错了之后的全部返工。花 20 分钟讨论，可能省掉 2 小时的无效开发。这个模式在 L3（跨层集成）和 L4（架构特性）的场景中尤其重要。
模式 F

先出多版视觉方案

"先做几版不同的视觉方案，在单独的 html，讨论讨论"

场景：涉及 UI/视觉设计的决策时，文字讨论很难对齐审美，不如直接看效果。

做法：让 Agent 在独立的 HTML 文件中生成多个方案，我直接在浏览器中对比选择。

真实案例
  • 追问功能浮动岛设计：Agent 生成了 6 种不同风格的浮动岛 HTML 预览——气泡式、工具条式、圆形按钮式等。我在浏览器中逐个看效果，最终选了"截图标注式"的方案。
  • 条款卡片配色方案：teal 系 vs blue 系 vs 渐变系。Agent 直接出了 3 份 HTML，每份里面有真实数据填充，不是空壳 mockup。
  • HTML 报表模板风格：Agent 生成了 8 种风格的 HTML 报表预览 → 我挑选了最符合品牌调性的那版 → 确认后才封装成组件。
价值：用"看"代替"想"。审美类的决策不适合纯文字讨论，直接看效果能节省大量来回沟通的时间。这也是 AI 作为"低成本试错引擎"的核心场景。
模式 G

文档补充与已知问题记录

"文档补充一下，把需求场景重点补充，目前还有局限或采用临时方案或已识别到问题但决定暂时不改的，都明确列一下"

场景：功能做完后，需要给同事或未来的自己留下清晰的上下文——不只是"做了什么"，还有"没做什么"和"为什么没做"。

做法：让 Agent 回溯整个开发过程，将讨论中提到的局限、临时方案、已知问题等系统性地整理到文档中。

真实案例
  • 头像功能文档：功能代码完整，但本地无法端到端测试（LOCAL 模式下系统 DataItem 不持久化到 MySQL）。Agent 记录了：(1) 前后端代码实现 (2) 本地测试被阻塞的原因 (3) 需在 Test 环境验证 (4) DataItem 路由机制的技术细节。
  • 24 份实施文档：每个 feature 完成后，Agent 自动生成一份文档——需求背景、实现方案、修改文件列表、已知局限、后续计划。直接作为 code review 材料。
  • P2 渐变色决策记录：明确记录了"4 处渐变需要讨论"和"周一找擦神对齐"——不是遗忘，而是有意识地推迟决策。
价值：文档是给人看的。把"我知道但没做"和"为什么没做"写清楚，比光写"做了什么"重要得多。这是你自己手写绝对不会做的事情——让 AI 做你嫌麻烦的事。

核心心得：不要只用 Agent 写代码。它的思考、审查、解释、文档能力同样强大。真正的 vibe coding 是全方位的协作，不是单方面的指令下达。

2.5 效率对比：假设传统开发模式

传统开发模式（估算）

  • 总耗时 ~120+ 小时（3~4 周）
  • 需要提前写完整 PRD
  • 文档沟通，容易理解偏差
  • 跨仓库需多人沟通对齐
  • 文档通常不会这么详细
  • 方向调整成本较高

Vibe Coding 模式

  • 总耗时 ~48 小时（5 天）
  • 对话中实时对齐需求
  • 截图 + 实时反馈，2~3 轮收敛
  • 一个 Agent 理解全链路
  • 24 份文档自动生成
  • 方向调整成本极低（分钟级）
注意：这个对比不是说 vibe coding 比传统开发"好"。传统开发有代码评审、测试覆盖、团队协作等 vibe coding 目前不具备的优势。这个对比只是说明：对于探索性开发和原型验证，vibe coding 的效率优势非常明显。

2.6 关键发现

1

55% 的时间是 Agent 在工作，我在"等 + 看"

实际人力投入约 ~22 小时（讨论 10h + 测试 12h），产出了 60 个文件、24 份文档。人均效率极高，但前提是你知道该让 Agent 做什么。

2

测试反馈的 25% 不可压缩

无论 Agent 多强，你都需要自己验证效果。特别是 UI 类改动，"好不好看"只有人能判断。审美类调整（配色、间距）是测试阶段耗时最多的子项。

3

讨论探索的 20% 是 ROI 最高的投入

花 20 分钟讨论方案 vs 花 2 小时返工，选哪个很明显。L4~L5 的讨论时间更长，但正是这些讨论避免了大量无效开发。

4

"无效投入"约占 5~8%

头像功能本地环境搭建（~1.5h）、P2 渐变色来回调整（~1h）、Agent 过度工程后的回退（零散）。这个比例已经很低了——传统开发中的无效投入通常高得多。

2.7 重要说明：从开发完成到生产可用

以上耗时仅覆盖"开发实现"阶段

要真正上线生产环境，后续还有不可忽视的工作量。

阶段内容说明
Code Review同事审查代码质量、架构合理性、安全风险视改动量而定
测试环境部署合并分支、部署到 Test 环境、配置联调包含环境问题排查
测试验证功能测试、回归测试、边界场景覆盖特别是跨仓库联动的集成测试
调优反馈测试中发现的问题修复、UI 微调、性能优化通常需要 2~3 轮
产品验收产品经理/设计师验收，确认符合预期可能触发新一轮调整

这部分工作量不应该被低估。Vibe Coding 极大地压缩了"从想法到可运行 Demo"的周期，但从 Demo 到生产可用之间的 Review → Test → Fix → 验收 流程，仍然需要团队协作和时间投入。

换句话说：Vibe Coding 改变的是"开发实现"这个环节的效率，而不是整个软件工程流程的效率。后续环节的工作量取决于团队的工程规范和质量要求。
