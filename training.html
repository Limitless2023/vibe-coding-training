二、故事线：从一个间距到全栈开发

2.1 动机源头

上周五晚上，我第一次拿到了生产环境的代码权限。

动机非常简单：MeCheck 上面接了 MeAgent，但它的 Markdown 输出正文行间距非常大，我觉得可以优化一版，让间距变小一些。

虽然这个调整可以让 AI 试一下效果，但如果我单独在 AI 里调，可能看不出直观的反馈。所以我就想着能不能直接在源代码上调，直接在系统中看效果。出于这个动机，我找了擦神申请权限，周五晚上拿到了代码。

2.2 第一步：建立认知

拿到代码后，我没有急着改 UI，而是先让 Agent 帮我做了一个全面的架构分析。两个动机：

  1. 售前需要 — 我们经常需要介绍技术架构，之前听过培训，但名词相对偏生硬。既然有了源代码，让 Agent 从中提炼，理解会更深。
  2. 视觉产出 — AI 工具的流程图语法非常好，画出来的图很漂亮，直接可以用于 PPT 汇报。

2.3 小步探索 → 正反馈循环启动

在建立认知之后，我回到了最初的动机 —— 调整 Markdown 展示 UI。

小需求
→
Agent 改代码
→
系统里看效果
→
满意？不满意→截图反馈→循环
满意→下一个需求

关键转折点：反馈极其直观且快速。传统模式下这个循环可能要几天（写需求 → 排期 → 开发 → 验证），现在几分钟就能看到效果。

2.4 想法打开：传统模式的痛 vs 新模式的优势

随着第一个 Feature 改好，我的想法逐渐增加了。我很好奇 Agent 还能帮我做什么。

Case 3：Sub-Agent 实时可视化 — 技术上最复杂 3 仓库

展示「从 0 到 1 的全栈 Feature」—— 需求分析 → 方案设计 → 开发实现 → 测试调优

需求场景：

  • 之前：子 Agent 运行完全是黑箱，只能看到最终结果
  • 之后：实时看到每个子 Agent 的思考 / 工具调用 / 文本输出

方案设计（和 Agent 一起完成）：AI 出了 3 个方案：

  • 方案 A：纯前端轮询
  • 方案 B：后端事件流 + 前端实时渲染 ← 选了这个
  • 方案 C：WebSocket 双工

选完后，AI 拆成 3 个独立阶段（后端事件流 / 前端 store / 组件渲染），每阶段单独验证。这就是"复杂度是怎么被管理的"。

开发实现（横跨 3 个仓库）：

  1. meagent 框架层（Python）— _subagent_events.py：ContextVar 事件队列；_spawn_subagent.py：运行时发射 start / thinking / tool_call / text / end 事件

四、真实工作流回放

每一个功能改动，都经历四个阶段。我全程做的事情：描述、选择、判断、验收。
提需求 & 探索
→
协作制定方案
→
开发 & 验证
→
完成 & 归档

阶段 1：提需求 & 探索

核心原则：截图 > 文字

一张截图 + 圈出问题，比写 500 字描述精准 10 倍。

场景 A — 我知道要什么

直接截图 + 圈出 + 一句话。例："这个间距太大了"、"错误了怎么不是红色的？" — AI 秒懂，不需要来回对齐。

场景 B — 我有方向但不确定细节

给 AI 一个参考系，让它理解设计语言。例："参考 Vercel AI 的风格"、"shadcn 那种扁平极简" — AI 会去分析参考对象的设计模式，而不是瞎猜。

场景 C — 我只有模糊想法

让 AI 先探索，调研最佳实践、分析现状。例："子 Agent 运行过程现在看不到，调研一下其他产品怎么做的" — AI 返回几种方案 + 各自优劣 → 我再做选择。

场景 D — 我发现了问题

截图 + 对比预期行为。例："这里工具调用显示 JSON，但应该显示中文意图"、"报错了但没红色提示"

阶段 2：协作制定方案

核心原则：简单的直接做，复杂的先对齐。

简单任务

改间距、调颜色 → 跳过 plan，直接执行。1 分钟。

中等任务

多文件改动 → AI 出方案，我确认后执行。

复杂任务

跨仓库/架构 → 多方案选择 + 分阶段拆解。

有歧义

多仓库联动 → 先出 plan，对齐理解再动手。

真实案例 — 子 Agent 可视化（高复杂度）：AI 出了 3 个方案（纯前端轮询 / 后端事件流 + 前端实时渲染 / WebSocket 双工）→ 我选了方案 B → AI 拆成 3 个独立阶段 → 每阶段单独验证。

阶段 3：开发 & 验证

核心原则：截图驱动反馈循环，通常 2-3 轮收敛。

场景 A — 单任务迭代（最常见）

AI 改完 → 我刷新看效果 → 截图反馈 → AI 继续改。

例："错误了怎么不是红色的？" → AI 追查全链路 → 发现 normalizeToolResult 漏检了错误格式 → 改 2 个文件 → 刷新 → 红了 → 我说"上面那个 badge 多余" → AI 再删掉 → 完美。从发现问题到修好，15 分钟。

场景 B — 多 Agent 并行

同时开 2-3 个 Agent 做不同功能，约定好各自改哪些文件。本质上和管理多个开发者分配任务没什么区别。

场景 C — 跨仓库联动

Purpose 功能跨 3 仓库 — copilot 加参数 → meagent 注入 schema → frontend 取值展示。如果只改了后端但前端不知道怎么取，就白做了。AI 能理解这条链路。

场景 D — 踩坑与调试

React 状态不更新：AI 自己定位到是数组引用没变，改用新数组触发渲染
代理变量干扰：ALL_PROXY 和 all_proxy 都会被 httpx 捡到，大小写都要清
ESLint 规则差异：项目配置不认 _ 前缀的 unused 变量 → AI 第一次踩了坑，记住后再没犯过

阶段 4：完成 & 归档

核心原则：每个功能做完不是结束，归档才是结束。

  • 写改动文档：每个功能一份实施文档，最终产出 24 份文档，直接能作为 code review 材料。这是我自己手写绝对不会做的事情。
  • 结构化提炼需求：把散落在对话中的需求整理成结构化文档
  • 准备提交 & Code Review：AI 帮整理 commit 历史、写 review 请求、标注重点

五、复杂度地图 — 什么 AI 能独立干

L1 CSS/样式~100% AI
L2 组件重构~90% AI
L3 跨层集成~70% AI
L4 架构特性~50% AI
L5 超难问题换 Agent
复杂度占比AI 自主度我做什么举例
L1 CSS / 样式~25%~100%说一句话字号统一、间距、行高
L2 组件重构~25%~90%确认方案Thinking 折叠、复制按钮、错误态红色
L3 跨层集成~20%~70%做设计决策Purpose 全栈（3 仓库）、Avatar 全栈
L4 架构特性~15%~50%选方案 + 拆阶段子 Agent 事件流、监控弹窗、Artifact
L5 超难问题~15%单 Agent 搞不定换 Agent + 调参数子 Agent 回放显示机制

L1-L4：同一个 Agent 内闭环

  • L1-L2 基本上"说一句话就搞定"，AI 自主度极高
  • L3 需要你理解系统边界（哪些仓库、哪层负责什么），做出设计决策
  • L4 需要你选方案 + 拆阶段，但每个阶段内 AI 仍然能独立完成

结论：L1-L4，人的价值在于方向和判断，AI 负责执行。

L5：单 Agent 的极限 — 换视角破局

备注：这里的"超难"是对我来讲。我没有编程背景，无法自己介入代码逻辑去辅助 AI。同样的问题对有经验的工程师来说可能并不难——但这恰恰说明了一个 vibe coder 的天花板在哪里，以及怎么用策略去突破它。

有些问题真的很难：Agent 连续改了两三轮，逻辑还是乱七八糟。

我遇到的真实场景：子 Agent 历史回放的显示机制——涉及事件持久化、状态恢复、组件渲染时序，一个 Agent 怎么改都不对。

破局方法：换一个不同视角的 Agent

从 Claude Opus 4.6 → 切换到 Codex 5.4
Thinking level 拉到最高（超高）
让它从头审视问题，重新思考 → 约 1 小时，完美解决。

为什么有效？

  • 不同模型有不同的"思维方式"——同一个问题，换个模型可能一下子就想通了
  • 高 thinking level = 更深的推理链，适合复杂逻辑问题
  • 关键不是"AI 不行"，而是"这个 AI 在这个配置下不行"——调 Agent 就像调团队成员

L5 的启示：当一个 Agent 连续 2-3 轮没进展 → 不要死磕，换一个。你的判断力在于：知道什么时候该换人。

六、心得 & 踩坑

6.1 最佳实践

做法说明
1. 截图 > 文字一张截图 + 圈出问题，比写 500 字精准 10 倍。真实案例：工具报错没红色提示 → 截图圈出 → AI 15 分钟全链路修好
2. 给参考系"参考 Vercel AI 的风格"比"帮我优化 UI"有效太多。AI 会分析参考对象的设计语言
3. 增量推进P0 → P1 → P1.5 → P2 → …，不要一次给 AI 太大的目标。中间任何一步不对都能立刻纠正
4. plan → confirm → execute让 AI 先出计划 → 你确认 → 再执行。避免 AI 冲去改了一堆你不想改的东西
5. 先看现状再动手每次改动前，先让 AI 读当前代码。Purpose 功能让 AI 先把 3 仓库全部读一遍再动手
6. 让 AI 写文档每个改动完成后 AI 自动写实施文档。24 份文档，直接作为 code review 材料。你自己绝对不会写
7. 多 Agent 并行一个做 A 功能，另一个做 B 功能，约定好文件边界 = 管理多个开发者

6.2 AI 做得好的地方

全栈一次性打通

Purpose 跨 3 仓库、20+ 个文件，AI 理解了整条调用链路，一次性打通。

CSS 手术刀式修改

精准找到该改哪个 className、改到什么值，不影响其他组件。真的理解了样式继承链。

错误追查全链路

从截图到修好 15 分钟，跨 4 个文件的调用链。我只截了一张图说"这个不对"。

"先预览后动手"

AI 先生成 8 种风格 HTML 预览 → 我挑选 → 确认后才封装。先看到成品再写代码。

5. 决定"不做什么"— 用第一性原理做减法

有一个需求看起来很合理：在用户输入框加一个"优化 prompt"的按钮，让 AI 先帮用户润色、补全 prompt，再发送给 Agent。

我和 Agent 一起评估后决定不做。回到第一性原理：问题的本质是"Agent 能不能理解用户意图"，而不是"用户的 prompt 写得好不好"。

方案做法判断
❌ prompt 优化按钮用户写完 → 点按钮 → LLM 润色 → 再发送多一步操作、多一次 LLM 调用、本质上是把责任推给用户
✅ Agent 主动澄清Agent 理解能力强，遇到模糊输入主动追问把复杂度留在 Agent 侧，用户体验更自然
Agent 足够聪明就不需要用户自己打磨 prompt——这正是 Agent 区别于传统 chatbot 的地方。不是所有"看起来有用的功能"都该做，做对决策比多做功能更重要。

6.3 踩过的坑

坑 1：UI 调整与设计系统缺失

最开始我不确定 MeFlow 是否有一个完整的设计系统，在代码仓库中也没看到可以直接复用的 UI 样式规范。所以当时采取的是"小步快跑"的模式，直接给 AI 描述或截图，逐步修改。

后来和 AI 共同打磨了一下，目前 MeAgent 这块的 UI 样式和配色我觉得还算可以。但过程中确实来回很多轮，效率不算高。

周一我找擦神汇报探索成果时，深有感触地说：无论 MeFlow 还是 MeCheck，我们都需要一个完整的设计系统。有了设计系统，UI 的修改和规范化落地效率会提升非常多。

教训：工具链很重要（浏览器插件定位元素，指哪改哪）；设计系统是基础设施（统一 Design Token，AI 有规范可循）

坑 2：本地全栈环境的天坑

做头像上传功能时，我想尝试全本地联调。成功把后端跑起来了，但没考虑到：要完整运行 MeFlow，还需要整个系统的初始化数据（用户、角色、权限、租户配置等等）。这些我本地根本没有。

结果让 AI 花了一两个小时在那里安装环境、尝试各种初始化方式，实际上做了无用功。最后果断放弃，完整记录了代码实现，撰写文档说明模块设计。

教训：AI 很擅长写代码，但不了解你的基础设施现状。该问人的时候要问人，不要让 AI 在一个它解决不了的问题上空转。

回过头看其实也还不错，只是浪费了一两个小时的 token，起码学到了东西。

更多踩坑速记

坑教训
不要说"帮我优化"太模糊，越具体结果越好，越模糊返工越多
AI 有时候过度工程改一个间距顺手"优化"旁边组件 → 引入新 bug。要说"只改我说的"
代理变量大小写ALL_PROXY 清了但 all_proxy 还在 → httpx 捡小写的 → 超时
审美类调整容易收不了尾渐变色调了半天。要设一个"够了"的标准，不然无限循环

七、不止写代码

7.1 需求、原型与设计探索 — AI 是你的低成本试错引擎

核心观点：有想法但拿不准时，让 AI 以极低成本快速出多个方案，你只需要选。

模糊想法
→
AI 出多个方案
→
对比选择
→
确认后实现
  • icon 选型：工具卡片需要图标 → AI 生成了好几套方案 → 对比后选最合适的。自己找图标库逐个比对至少半小时，AI 几分钟出完。
  • 配色方案：品牌色 #13ACBA，需要搭配色板 → AI 从代码提取已有色值 → 生成完整色板。进一步生成了 8 种 HTML 报表模板风格。
  • 排版参考：我说"参考 Vercel AI 的风格"，AI 分析了 Vercel 的设计语言——大量留白、极小字号差、几何图标——然后映射到我们的组件上。
  • 子 Agent 监控弹窗布局：1 个 Agent 怎么排？2 个怎么排？5 个以上？AI 直接出了布局规则表 → 确认 → 实现。

7.2 文档、回溯与经验整理

核心观点：你现在看到的这份分享文档，就是 AI 帮我编写和整理的。

  • 改动文档：24 份文档，记录了做了什么、为什么、怎么验证。直接成为 code review 参考材料。
  • 开发回溯：让 AI 遍历所有 commit 历史 + 对话记录，生成完整开发小结。
  • 经验提炼：这份分享大纲从 v1 到 v4，每一版都是"我提方向 + AI 执行"。
  • Code Review 请求：包含 4 个仓库的分支名、commit 列表、每个改动的重点标注——也是 AI 帮准备的。

7.3 架构分析与材料产出

  • 让 Agent 从源码中提炼 MeAgent 技术架构，产出了完整的架构材料
  • 生成了可直接用于 PPT 汇报的流程图和对比表
  • 作为售前，这些材料比听培训记笔记效果好太多

八、为什么这套能 work

8.1 前置条件：我一直在用 AI

这不是我第一次用 AI 做项目。

  • 大约半年前开始用 AI 做项目，有些做得不怎么样，中途半途而废了
  • 去年年底 + 春节期间，明显感受到模型能力的跃升——能做的事情质变了
  • 春节期间，出于好奇心，用 Claude Code + Opus 4.6 从 0 到 1 构建了一个 AI Agent（可观测性、Memory、上下文压缩、Skill/MCP 机制）
  • 后来一个周末突发奇想，把它做成了跨平台 App（手机 + 电脑 + Web）

到了这次 Copilot UI 改进时，我已经对 AI Agent 的内部机制有了直觉。虽然我不写代码，但我知道该让 AI 做什么、不该做什么、做到什么程度算完成。

8.2 模型能力的跃升

我每天都在感叹 AI 又帮我做了什么新东西。
  • 从"能写简单函数"到"能理解 4 个仓库之间的调用关系并正确修改"
  • 从"给它一个文件改"到"它自己搜索代码库、定位问题、跨文件修复"
  • AI 在成长，用 AI 的人也需要跟着成长——用法、期待值、协作方式都在变

8.3 结果的可验证性

Coding 有一个特别好的点：可验证。

AI 写完代码
→

→

→

→
↩ 循环
  • AI 写完代码 → 连接环境 → 我能立马看到效果
  • 这次改的大部分是前端交互——UI 对不对，刷新一下就知道
  • 写文章？好不好很主观。做设计？审美难量化。写代码？跑一下就知道对不对。

8.4 开发规范：Claude.md / AGENTS.md

我有一个项目级的规范文件（Claude Code 中是 CLAUDE.md，Codex 中是 AGENTS.md）。

效果：AI 进入任何仓库前，先读 CLAUDE.md → 理解项目结构、工程原则、沟通方式 → 然后才动手。这就像给新员工一份 onboarding 文档——规范越清晰，AI 产出越稳定。

8.5 心态：AI 比我聪明

这是一个非常大的前提。

当 AI 做不好某个东西时，我不会怀疑是它的能力问题。我会反过来想：

上下文没给够

补充相关文件、背景、约束

表述不清楚

换说法、截图标注

任务太大

拆成更小的阶段

该换人了

换模型、调参数

我很相信 AI 啥都能做。 虽然实际情况不一定完全如此，但这种信任让我可以非常放心大胆地去探索。

结果就是：每一个环节都在给我正反馈。

  • AI 做对了 → 验证通过 → 信心增强 → 给它更难的任务
  • AI 做错了 → 我调整上下文/表述 → AI 做对了 → 我学会了怎么更好地协作
  • 进入一个正向循环
如果你一开始就不信，你不会给它机会 → 它就真的做不了 → 你更不信。
Vibe coding 的起点是信任。

九、我最大的几个感受

1. 好奇心

我是真的很好奇 AI 能做什么、不能做什么。这种好奇心驱动我去尝试各种各样的事情——有些成了，有些废了，但每一次都在积累经验。

2. Get your feet wet

擦神给的建议：先下水试试。不用一上来就做大项目，随便找一个小问题让 AI 帮你改——你会发现门槛真的没那么高。

3. Get your hands dirty

不要怕做一些 dirty work。AI 帮你省了 80% 的苦力，但剩下的 20% 还是得自己扛。你负责"开门"，AI 负责"装修"。

4. 多多记录并分享输出

记录让你复盘，分享让你整理。非常感谢擦神邀请我来做这次分享——借这个机会，我也和 AI 一起重新梳理了自己的实践心得与不足。

「想到就能做到」的周期极短

以前：想到一个改进 → 记到 backlog → 排期 → 等开发 → 上线
现在：想到 → 5 分钟内 AI 在改 → 15 分钟后看到效果
这个反馈速度彻底改变了工作节奏。
